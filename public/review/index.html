Decisions we need your input on

Five choices that unblock the rest of the project.

1. How should members log in?

HBS Women's Association uses "Sign in with Harvard Key" — but they get it via a paid alumni platform (iModules), not by building it. Harvard Key directly needs Harvard IT approval and can't run on our simple hosting.

  • Defer login for v1 — launch the public site now, add login later. Recommended
  • Lighter verification — email magic-link or "Sign in with Google" + manual approval. No Harvard IT needed; captures more member data.
  • Harvard Key directly — strongest, but needs an official Harvard sponsor + IT approval (slow, uncertain).
  • Paid alumni platform — like HBSWA; handles login + CRM but costs money and replaces much of this custom build.

Your input: is member login required for launch, or a fast-follow? Is there a Harvard sponsor who could pursue Harvard Key?

2. Events page layout

The current build is a clean Luma-calendar-first page. The hbswa.org example is a richer events directory.

  • Keep Luma-first + add Webinars/Podcasts sections — less work. Recommended
  • Match the hbswa directory layout — closer to the example, more design work.

Your input: which layout do you prefer?

3. Domain (harvardintech.com)

We move the domain from Strikingly to the new site with zero downtime (stage on a subdomain, then switch). The only blocker is access.

  • Who controls the domain today? — the registrar login + DNS settings.
  • Is there email on @harvardintech.com? — if so we must preserve it during the move.
  • When do we cut over? — after everyone approves the staged preview.

Your input: point us to whoever manages the domain, and a target launch window.

4. How private should the preview be?

This page (and the work-in-progress site) are shared via a link. Free GitHub Pages links are public by nature.

  • Unlisted link + noindex + passphrase (what you're using now) — simple, free; a deterrent, not bank-grade security.
  • Real password protection — host the preview on Cloudflare/Netlify (free) for a true password gate.

Your input: is the current passphrase gate enough, or do we want real access control?

5. Who provides the content?

The site is built but empty in places. Filling it is point-and-click in the editor (no coding).

  • Chapter committees + events for Boston, NYC, London, DC/DMV, Seattle.
  • Board of Directors — real bios + photos.
  • Content Hub — blog posts, podcasts, newsletter links.

Your input: who owns gathering this, and by when?
