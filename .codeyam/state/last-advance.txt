Checklist:
[ ] Audit leftover dirty state and probe remote setup: `codeyam-editor editor commit-state`
    This scans `git status`, partitions dirty files into:
      - `.codeyam/` leftovers (regenerated caches, merge-driver files, etc.) —
        silently committed as a separate `chore: update codeyam state` commit
      - external leftovers (anything outside `.codeyam/`) — returned in the
        `external` array for you to surface to the user
    `commit-state` also runs `session-cleanup` (orphan screenshots, log rotation,
    tmp prune) per `editor.json::cleanup` BEFORE the scan, so any deletions land
    in the chore commit. Best-effort; failures log to stderr.
    Refuses to commit if any `.codeyam/` JSON file is unparseable — fix it or
    regenerate it before retrying.
    **`.gitignore` is the authoritative skip-list. Files outside `.codeyam/`
    that survive `.gitignore` are project state and should be committed by
    default — do not prompt for untracked non-ignored files.**

    The JSON envelope ALSO carries a `remoteSetup` block:
      `{state, remoteName, ghInstalled, ghAuthenticated, suggestedDefaultRepoName}`
    Read these structured fields rather than re-probing remote / `gh` state
    yourself. Surfacing raw shell-probe output at the user is what this
    checklist exists to retire.

[ ] If the `external` array from `commit-state` is non-empty, partition it
    by `status` and act on each bucket separately:

    - `untracked` (status == "untracked") — auto-commit with NO prompt.
      Stage the paths and create a single `chore: include <paths>` commit.
      Add a one-line note in your user-facing summary so the user can see
      what got swept in (e.g. "Also committed:
      `.claude/scheduled_tasks.lock` (untracked, not gitignored).").

    - `modified` (any status other than "untracked" — modifications to
      existing tracked files) — these *might* be in-progress work from
      another session, so ask before acting. The `external[i]` entries
      already carry `path`, `added`, `removed`, and `summary` — read
      those structured fields rather than re-running a per-file diff
      probe in chat. Then use `AskUserQuestion` with a single combined prompt for
      the whole batch (not per file):

        "These are modifications to tracked files outside the work we
         just did:
         - <path> — +<added>/-<removed> (<summary>)
         - ...
         Should I commit them with the feature?"

      Options:
        Option 1: "Commit them now with the feature (Recommended)" →
                  stage the paths and create a `chore: include <paths>`
                  commit
        Option 2: "Leave them alone — part of other work in progress" → skip
        Option 3: "Discard these changes" → `git checkout -- <paths>` (confirm first)

    Leftover-file decisions and remote-setup decisions MUST fire as separate
    `AskUserQuestion` calls. Bundling them mixes "what should this commit
    contain?" with "should this project go on GitHub?" — two unrelated
    mental models that confuse non-technical users. Wait for the
    leftover-file answer before moving on to the remote-setup branch below.

[ ] Branch on `remoteSetup.state` from the `commit-state` envelope:

    - `remote-configured` → offer to push (AskUserQuestion — STOP and wait for the user's answer before proceeding):
      ask (AskUserQuestion): "Would you like me to push this commit to the remote?" with options "Yes, push" and "No, skip"
      If the user says yes, run: `codeyam-editor editor push`
      IMPORTANT: Use `codeyam-editor editor push`, NOT raw `git push`.
      Fast Commit: pushing a deferred commit to THIS feature branch is
      routine — `editor push` ships it without the emergency
      `--allow-deferred --reason` ceremony and prints a calm "Fast mode:
      pushing N deferred commit(s) ... finalize owed before merge to
      <primary>" note. (Pushing deferred commits directly to the primary
      branch is still a hard block — the "no un-finalized work on
      primary" invariant moves to the merge boundary, where the owed
      `session-finalize` clears it.)
      If the push output shows `"pulled": true`, new code was pulled via rebase.
      In that case, run `codeyam-editor editor post-merge-drift-sweep` BEFORE
      refreshing tests (canonical sequence: reconcile-registry + reconcile-
      glossary + audit; work through any `remainingActions`). Then refresh the
      test cache: `codeyam-editor editor refresh-tests`.
      (Fast Commit defers this drift sweep + refresh to the eventual
      `session-finalize` too — if you are in fast mode, a `"pulled": true`
      push just proceeds; the sweep is owed at finalize, not here.)
      If tests fail after the pull, fix the issues, commit, and re-push.
      (For the longer rationale, see
      `codeyam-editor editor step --slug post-merge-drift-sweep`.)
      (The canonical project config is `.codeyam/editor.json` — `.codeyam/config.json` is a stale artifact from older scaffolders, do not read or write it; the `no-stale-config-json` static check fails the build if it reappears.)

    - ANY no-remote state (`no-remote-ready`, `no-remote-gh-missing`,
      `no-remote-gh-unauthed`) → first check `remoteSetup.shouldRemind`:
      • If `false`, the user recently declined a backup — stay silent and
        proceed. Do NOT fire `AskUserQuestion`.
      • If `true`, ask the backup question (AskUserQuestion — STOP and wait for the user's answer before proceeding):
        ask (AskUserQuestion): "Your work is saved only on this computer right now — it isn't backed up anywhere. Want me to back it up on GitHub so you won't lose it if anything happens to this computer, and can open it from another computer or share it with others?" with options "Back it up on GitHub — private (only me) (Recommended)", "Back it up on GitHub — public (anyone can see)", and "Keep it on my computer for now"
      `remoteSetup.state` does NOT decide whether to ask — it only decides how
      to fulfill a "back it up" choice (GitHub is reachable over plain `git`
      even without the `gh` CLI, so every no-remote user gets asked):

        · `no-remote-ready` (gh installed + signed in) → ask `AskUserQuestion`
          for the project name on GitHub, defaulting to
          `remoteSetup.suggestedDefaultRepoName` (with "Other" for a custom
          name). Then run:
          `codeyam-editor editor github-setup --visibility {private|public} --name <name>`
          Use the helper, NOT raw `gh repo create`. Surface the helper's `url`.

        · `no-remote-gh-missing` / `no-remote-gh-unauthed` (no `gh` CLI) → use
          the guided path; nothing to install:
          1. Tell the user to create a new EMPTY repository at
             https://github.com/new (no README or .gitignore, so the first
             push isn't rejected) and copy the URL it shows.
          2. When they paste it, run:
             `codeyam-editor editor connect-remote --url <url>`
          3. Surface the resulting repo URL. You MAY add one line that
             installing the GitHub CLI (https://cli.github.com) would let this
             happen automatically next time — do NOT print an OS-specific
             install command.

      On "Keep it on my computer for now" (any no-remote state): acknowledge in
      one short line that the work stays only on this computer and isn't backed
      up to GitHub yet, run `codeyam-editor editor remote-backup-defer` to
      record the choice, and proceed. Do NOT ask a follow-up name question.
